## 논의 요약

### 1. 프로덕트 컨셉 정의

- 컨텐츠 소비 피드백 기반 취향 분석 & 추천 앱
- 장기 목표는 도서/영화/뮤지컬/전시 전 카테고리지만, **도서로 MVP 시작**
- 타겟: 자기 취향을 잘 모르는 일반 유저 ("베스트셀러만 읽는 시대에 취향을 찾아가는 즐거움")

### 2. 핵심 가치 두 가지

1. **서재 경험** — 책을 꽂아가는 뿌듯함 (리텐션 드라이버). 단, 서재 자체가 목적이 아니라 추천을 위한 데이터 수집 수단.
2. **취향 기반 추천** — 쌓인 피드백으로 맞춤 추천 (핵심 가치)

Eden의 포인트: 서재가 데이터 수집 수단이지만, 다른 서재 앱과 차별화된 감성적 경험이 있어야 한다.

### 3. 서재 차별화 아이디어

- 표지 색상 추출 → 책등 자동 생성 (유저만의 컬러 팔레트)
- 책 추가 시 "톡" 하고 꽂히는 애니메이션
- 마일스톤: 권수에 따라 서재 배경 업그레이드 (작은 선반 → 도서관)
- 책등 너비 = 페이지 수에 비례

### 4. 추천 엔진 방향

3가지 방식 검토:
- A. Content-based (태그 기반) → 태그 설계 부담
- B. Collaborative filtering → 유저 부족 시 작동 안 함
- **C. 벡터 유사도 채택** → 태그 불필요, 자연어 피드백이 곧 취향 벡터

→ 상세: `decisions/001-recommendation-engine.md`

### 5. 피드백 수집 설계

- 자유 텍스트만 받으면 정보량 부족 ("좋았어", "재밌었어")
- **가이드 질문 + 자유 텍스트 조합**으로 해결
  - "어떤 점이 좋았어?" → 선택지 (캐릭터/문체/세계관/스토리/메시지/분위기)
  - 선택 후 "구체적으로?" → 자유 텍스트
- "별로였던 것"도 수집 (negative signal)

### 6. 콜드 스타트 문제 & 온보딩

일반 유저는 읽은 책이 3~5권일 수 있음 → 서재가 텅 비어서 이탈 우려

해결:
- 온보딩에서 유명 책 그리드로 빠르게 등록 (탭탭탭)
- "읽고 싶은 책"도 서재에 포함
- 5권 + 피드백 3개부터 초기 추천 시작

온보딩 레퍼런스: Spotify (선택 쾌감), Letterboxd (포스터 비주얼), Blinkist (목적 기반)

### 7. 기술 스택 제안

Flutter + Supabase + 알라딘 API + OpenAI embedding + pgvector + FastAPI

→ 상세: `decisions/002-tech-stack.md`
