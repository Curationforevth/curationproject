# 추천 엔진 v3 설계 — 포인트 클라우드 방식

> 날짜: 2026-04-01
> 상태: 설계 완료, 212권 × 10 페르소나 실제 리뷰 기반 검증 통과 (10/10)

---

## 1. 핵심 원칙

**취향의 품질이 최우선이다.**

추천 엔진의 유일한 목적은 "유저가 다음에 읽을 책으로 이걸 추천받았을 때 '아 이거 내 취향이다'라고 느끼는 것"이다. 응답 시간, 확장성, 코드 구조는 모두 이 목적을 달성한 뒤에 의미가 있다.

### 1.1 취향이란 무엇인가

취향은 단일 속성이 아니라 **여러 벡터의 연결성으로 형성되는 공간**이다.

- 유저가 책에 남긴 **피드백**은 취향의 명시적 신호
- 유저가 좋아한 책의 **reason**(이 책을 좋아할 이유)은 취향의 축
- 책의 **description**은 전체 분위기
- 책의 **genre**는 장르 공간에서의 위치

이 네 가지는 모두 같은 임베딩 공간(text-embedding-3-large, 2000차원)에 존재하는 **점(벡터)**이다. 각각이 독립적인 점으로 존재하고, 점들 사이의 거리(코사인 유사도)가 취향의 연결성을 나타낸다. 벡터 자체를 합치거나 평균내지 않는다 — 각 점은 공간에서 독립적으로 유지된다. 최종 추천 스코어를 만들 때 점 간 거리의 가중합이 사용되지만, 이는 벡터 공간에서의 정보를 보존한 상태에서의 수치화이다 (섹션 4.1 참조).

### 1.2 취향 품질의 스펙트럼

유저의 입력량과 구체성에 따라 추천 품질이 달라진다.

| 입력 상태 | 추천 품질 | 설명 |
|-----------|----------|------|
| 좋아요 1~2권 | 방향만 잡힘 | 비슷한 장르 위주, 변별력 약함 |
| **좋아요 3권** | 방향 확실 | 장르+분위기 매칭 시작 |
| **좋아요 5권** | 신뢰 가능 | 모든 취향 유형(소설/실용서/과학)에서 안정 |
| 좋아요 5권 + 싫어요 1권 이상 | 변별력 강화 | 싫어하는 영역이 확실히 밀림 |
| 좋아요 5권 + 구체적 피드백 2개 이상 | 정밀 추천 | 피드백이 취향 축을 날카롭게 잡아줌 |

**온보딩 최소 조건: 좋아하는 책 5권.** 이 조건부터 추천을 시작한다. 피드백과 싫어요는 자연스럽게 쌓이도록 유도한다.

### 1.3 피드백 깊이와 취향의 관계

같은 책에 대해서도 피드백 깊이에 따라 생성되는 취향 벡터가 다르다.

**구체적 피드백** ("사회의 구조적 폭력을 범죄 서사로 파고드는 시선이 날카로웠다")
→ 피드백 자체가 강한 취향 벡터. 유저가 이 책의 **어떤 축**을 좋아했는지 명확.
→ 해당 축과 가까운 다른 책이 정밀하게 매칭됨.

**피드백 없음 (호오만)**
→ 피드백 벡터 없음.
→ 책의 reason/desc/genre 벡터만으로 취향 공간 형성.
→ 가장 넓은 추천. 피드백이 쌓일수록 좁혀짐.

참고: "좋았음", "별로" 같은 짧은 피드백도 임베딩되어 피드백 벡터로 사용된다. 정보량이 적어 넓은 매칭이 되지만, 피드백이 있는 것과 없는 것의 차이는 가중치 구조(H2)에서 자동 반영된다. 구현에서는 **피드백 있음/없음** 2가지로만 분기한다.

**싫어요**
→ 역방향 벡터. 해당 영역의 책을 밀어냄.
→ 구체적 싫어요 피드백은 특정 축만 밀어냄, 호오만 싫어요는 책 전체 영역을 밀어냄.

### 1.4 책이 가진 벡터와 유저 피드백 벡터의 관계

책과 유저의 벡터는 **같은 공간에 공존**한다. 합치거나 변환하지 않는다.

```
같은 임베딩 공간 (2000차원):

  유저의 점들:
    ● 피드백 "사회문제를 파고드는 시선이 날카로웠다"
    ● 피드백 "캐릭터 관계가 리얼해서 몰입됐다"
    ○ 좋아한 책A의 reason "범죄 피해자 가족의 고통을 통해 사법 시스템의 모순을 드러냄"
    ○ 좋아한 책A의 desc 벡터
    ○ 좋아한 책A의 genre 벡터
    ○ 좋아한 책B의 reason들...
    ✕ 싫어한 책C의 벡터들 (역방향)

  후보 책의 점들:
    ◆ reason "소외된 이웃의 삶을 담담하게 비추는 사회 고발적 시선"
    ◆ reason "다층 시점 전환으로 사건의 전모가 서서히 밝혀지는 구조"
    ◆ desc 벡터
    ◆ genre 벡터

  추천 = 유저의 점들과 후보 책 점들이 가까운 정도
```

유저의 피드백 벡터 ●가 후보 책의 reason 벡터 ◆과 가까우면 → 취향 매칭.
유저의 genre 벡터 ○가 후보 책의 genre 벡터 ◆과 가까우면 → 같은 장르 공간.
유저의 싫어요 벡터 ✕가 후보 책의 벡터와 가까우면 → 역방향으로 밀어냄.

**이 세 연결이 동시에 가까운 책이 강추천, 하나만 가까운 책은 약추천.** 각 점의 거리가 독립적으로 계산된 뒤 Weighted MaxSim으로 스코어가 만들어진다 (섹션 4.1 참조).

## 2. 아키텍처

```
┌─────────────┐     ┌──────────────────────┐     ┌─────────────┐
│ Flutter 앱   │────▶│ FastAPI 추천 서버       │────▶│ Supabase    │
│             │◀────│                      │     │ (저장만)     │
└─────────────┘     │  ┌────────────────┐  │     └─────────────┘
                    │  │ numpy 벡터 매트릭스│  │
                    │  │ (메모리, BLAS)   │  │
                    │  └────────────────┘  │
                    └──────────────────────┘
```

- **Supabase**: Auth, 책 데이터, 유저 데이터, 피드백 저장. 추천 연산 하지 않음.
- **FastAPI 추천 서버**: 벡터 검색, 추천 계산. 서버 시작 시 전체 벡터를 메모리에 로드.
- **numpy 행렬곱**: Weighted MaxSim은 점 구름 대 점 구름 비교이므로 단일 벡터 ANN(FAISS)으로 직접 할 수 없음. numpy BLAS 배치 행렬곱으로 처리. 2,000권 기준 ~20ms 추정. 규모 확장 시 MUVERA 스타일 압축 또는 pre-filtering 적용 가능.
- **Flutter 앱**: 온보딩, 피드백 입력, 추천 결과 표시.

### 2.1 왜 Supabase에서 추천하지 않는가

v1~v2에서 Supabase pgvector로 추천을 시도했으나:
- 2000차원 벡터 × 19,000건 CROSS JOIN → IO 고갈, 타임아웃
- Free tier Disk IO 한도에서 배치와 실시간이 공존 불가
- 유저 1,000명 동시 접속 시 DB가 버틸 수 없음

추천 연산을 별도 서버로 분리하면 Supabase는 저장만 담당하고, 벡터 검색은 numpy 행렬곱(메모리)으로 밀리초 단위 처리.

## 3. 데이터 모델

### 3.1 책 벡터 (사전 배치)

각 책은 다음 벡터를 가진다:

| 벡터 | 개수 | 생성 시점 | 설명 |
|------|------|----------|------|
| reason 벡터들 | 5~8개 | 배치 (LLM 추출 → 임베딩) | "이 책을 좋아할 이유" 각각의 벡터 |
| desc 벡터 | 1개 | 배치 (rich_description HTML 제거 후 최대 2000자 임베딩) | 책의 전체 분위기 벡터. 200자로는 마케팅 문구만 담기므로 2000자 필요 (테스트: 2000자가 정답1위 6/7, 200자는 5/7) |
| genre 벡터 | 1개 | 배치 (장르 문자열 임베딩) | 장르 공간에서의 위치 |

**reason 추출 프롬프트 핵심 규칙:**
- 15~40자의 구체적 명사구/설명구
- 이 책만의 맥락이 담겨야 함 (다른 책에 붙여넣을 수 없어야 함)
- 고유명사, 평론 표현, 범용 태그 제외
- 예: "범죄 피해자 가족의 고통을 통해 사법 시스템의 모순을 드러냄" (O)
- 예: "사회파 미스터리" (X — 범용 태그)

### 3.2 유저 취향 벡터 (실시간 생성)

유저가 책에 대해 입력할 때마다 취향 벡터가 생성된다.

| 입력 유형 | 생성되는 점들 | 각 점의 가중치 |
|-----------|-------------|-------------|
| 호 + 피드백 있음 | 피드백 벡터 1 + reason 벡터들 + desc 1 + genre 1 | 피드백: +3.0, reason: +0.5, desc: +0.3, genre: +0.3 |
| 호 + 피드백 없음 | reason 벡터들 + desc 1 + genre 1 | reason: +1.0, desc: +0.7, genre: +0.7 |
| 오 + 피드백 있음 | 피드백 벡터 1 + reason 벡터들 + desc 1 + genre 1 | 피드백: -3.0, reason: -0.5, desc: -0.3, genre: -0.3 |
| 오 + 피드백 없음 | reason 벡터들 + desc 1 + genre 1 | reason: -1.0, desc: -0.7, genre: -0.7 |

**가중치 — 212권 × Weighted MaxSim 검증 기준 초기값, 2,000권 검증 후 조정 가능.**

피드백이 있을 때: 피드백 벡터가 가장 강한 신호 (3.0), 나머지는 보조. 피드백이 유저의 취향을 가장 명확하게 나타내기 때문.
피드백이 없을 때: reason이 가장 강한 신호 (1.0), desc/genre가 보완 (0.7). 책의 벡터들이 취향 공간을 채움.

**가중치 근거:** 이전 H2 그리드 서치(576개 조합, 20 시나리오)에서 호/오 기본 가중치 도출. MaxSim 전환 시 피드백 벡터를 ×3.0(가장 강한 신호), reason을 ×0.5, desc/genre를 ×0.3(보조)으로 설정. 피드백 없을 때는 reason ×1.0, desc/genre ×0.7. 이 비율은 212권 10 페르소나 테스트 기반이나 체계적 그리드 서치는 아니므로, 2,000권 검증에서 재수행 권장.

## 4. 추천 알고리즘

### 4.1 Weighted MaxSim (ColBERT 기반)

IR(Information Retrieval) 분야에서 검증된 ColBERT의 late interaction 방식을 기반으로 한다. 각 점이 독립적으로 가장 가까운 상대 점을 찾아 매칭하며, 벡터 자체를 합치거나 평균내지 않는다.

```python
def weighted_maxsim(user_vecs, user_weights, book_vecs):
    """
    user_vecs: (M, D) 유저 점 구름 — 각 점은 독립적
    user_weights: (M,) 가중치 (+: 좋아요, -: 싫어요)
    book_vecs: (N, D) 후보 책 점 구름
    """
    # (M, N) 유사도 행렬 — 정규화된 벡터이므로 dot product = cosine sim
    sim_matrix = user_vecs @ book_vecs.T
    # 각 유저 점에서 후보 책의 가장 가까운 점과의 유사도
    max_sims = sim_matrix.max(axis=1)  # (M,)
    # 가중 스코어
    return float(user_weights @ max_sims)
```

**작동 방식:**
1. 유저의 각 점(벡터)이 후보 책의 모든 점과 유사도를 계산
2. 각 유저 점에서 후보 책의 **가장 가까운 점(MaxSim)**을 찾음
3. 각 MaxSim에 해당 점의 가중치를 곱함
4. 가중치가 곱해진 MaxSim들의 합이 최종 스코어

**한계:** 마지막에 가중합이 들어간다. 모든 점 구름 거리 메트릭(Hausdorff, Chamfer, EMD, MaxSim)이 최종적으로 하나의 숫자를 만들며, 이는 구조적으로 불가피하다. 다만 벡터 자체를 합치는 것(가중평균)과 점 간 거리의 가중합은 다르다 — 전자는 벡터 공간에서 정보를 잃지만, 후자는 각 점의 독립성을 보존한다.

**선택 근거 (deep research 결과):**
- Hausdorff: outlier 1개에 전체가 좌우됨 → 부적합
- Chamfer: min 기반이라 매칭 방향이 반대 → 차선
- EMD/Sinkhorn: 이론적 최적이나 O(n²) × iteration → 실시간 불가
- **MaxSim: BLAS 최적화로 ~20ms, ColBERT에서 검증됨, 가중치 자연스러움 → 채택**

### 4.2 왜 이 방식이 작동하는가

**카카오프렌즈(어린이 학습만화)가 미스터리 팬에게 추천되지 않는 이유:**
- 유저의 genre 벡터 "추리/미스터리소설"과 카카오프렌즈의 genre 벡터 "어린이/과학학습"이 멀다
- reason에서 "서바이벌 스토리"가 약간 겹쳐도, genre/desc 점들이 모두 멀어서 종합 스코어가 낮다
- 같은 미스터리 소설은 reason + genre + desc 모든 점이 가까워서 압도적으로 높다

**범용 피드백("세상 보는 눈이 바뀌었음")이 과학 독자에게 맞는 추천을 내는 이유:**
- 피드백 벡터 자체는 범용적이라 여러 장르에 걸림
- 하지만 유저가 읽은 과학책들의 genre 벡터, desc 벡터, reason 벡터가 과학 영역을 형성
- 후보 책 중 과학서는 이 영역의 모든 점과 가까움 → 스코어 높음

**취향 전환(실용서→소설)이 반영되는 이유:**
- 소설에 구체적 피드백(가중치 3.0)이 있으면 실용서 호오만(가중치 1.0)보다 강한 신호
- 다만 실용서 권수가 많으면 점 구름 크기로 역전 가능 — 이는 "많이 읽은 장르"가 취향의 더 큰 부분을 차지한다는 의미이므로 반드시 잘못된 것은 아님
- 2,000권 검증에서 이 균형을 재확인 필요

## 5. reason 추출 프롬프트

```
이 책을 좋아할 독자 관점에서 "좋아할 이유" 5~8개. 15~40자 명사구.
이 책만의 구체적 맥락. 고유명사/평론/범용태그 제외.

도서: {title} ({genre})
설명: {desc}  ← rich_description HTML 제거 후 최대 1500자

JSON: {"reasons": [...]}
```

**프롬프트 검증 기준:**
- 같은 reason이 다른 책에 그대로 붙여넣을 수 있으면 실패
- "성장", "감동", "몰입감" 같은 범용 태그 → 실패
- "범죄 피해자 가족의 고통을 통해 사법 시스템의 모순을 드러냄" → 통과

**검증 결과:**
- 6권 → 14권 → 85권 → 212권 단계적 테스트
- 유사도 1.000 문제(범용 태그) 완전 해결
- 유사도 그라데이션 0.3~0.6 범위에서 자연스러운 분포

## 6. 인프라

### 6.1 FastAPI 추천 서버

```
recommendation-server/
├── main.py              # FastAPI 앱
├── models.py            # 요청/응답 스키마
├── engine/
│   ├── loader.py        # 서버 시작 시 벡터 로드
│   ├── index.py         # 벡터 매트릭스 관리 (numpy BLAS)
│   └── recommender.py   # 포인트 클라우드 추천 로직
├── api/
│   ├── recommend.py     # GET /recommend/{user_id}
│   └── feedback.py      # POST /feedback
└── requirements.txt
```

**엔드포인트:**

| 엔드포인트 | 메서드 | 설명 |
|-----------|--------|------|
| `/recommend/{user_id}` | GET | 추천 목록 반환 (top N) |
| `/feedback` | POST | 피드백 수신 → 임베딩 → Supabase 저장 |
| `/health` | GET | 서버 상태 확인 |

**서버 시작 시:**
1. Supabase에서 전체 book_love_reasons + books 테이블 로드
2. reason/desc/genre 벡터를 메모리에 적재
3. numpy 행렬로 메모리 내 인덱스 구축

**피드백 수신 시:**
1. 피드백 텍스트 → OpenAI embedding API (1회, ~100ms)
2. Supabase에 피드백 저장
3. 메모리 내 유저 취향 벡터 업데이트

### 6.2 성능 목표

| 지표 | 목표 | 근거 |
|------|------|------|
| 추천 응답 | <100ms | 212권 numpy 행렬곱: 15~38ms 실측. 2,000권은 배치 행렬곱으로 ~20ms 추정 (벡터를 하나의 행렬로 묶어 단일 BLAS 호출) |
| 피드백 처리 | <500ms | OpenAI embedding ~100ms + DB 저장 |
| 동시 접속 | 초당 ~200 요청 (4 workers × 50 req/s) | 1,000 MAU 수준 충분. 진짜 동시 1,000명은 workers 수 증가 또는 비동기 처리 필요 |
| 메모리 사용 | <500MB | 2,000권 × ~10벡터 × 2000차원 × 4byte ≈ 160MB |

### 6.3 배포

- Render 또는 Fly.io (무료~$7/월)
- 환경변수: SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY, OPENAI_API_KEY

## 7. 데이터 마이그레이션

### 7.1 기존 데이터 처리

| 데이터 | 현재 상태 | 조치 |
|--------|----------|------|
| book_love_reasons 19,450건 | 범용 태그 ("사회파 미스터리") | 유지하되, 새 프롬프트로 보완 reason 추가 |
| book_embeddings | text-embedding-3-small 1536차원 | book-to-book 유사도용으로 유지 |
| books 테이블 | description, genre 존재 | desc/genre 임베딩 새로 생성 (text-embedding-3-large 2000차원) |

### 7.2 새로 수집할 데이터

1. **새 프롬프트 reason** — ~2,000권 × 5~8개 = ~12,000건
   - 기존 reason과 별도 컬럼 또는 source='v3_context_rich'로 구분
   - 50권 단위로 품질 체크 후 전체 배치
   - safe_rerun.py 활용 (insert 20행씩, chunk 간 2초 sleep)

2. **desc 임베딩** — ~2,000권 × 1개
3. **genre 임베딩** — ~2,000권 × 1개

## 8. 실패 기록과 교훈

v1~v2에서 8번의 실패를 거쳤다. 이 설계는 그 실패들을 직접 해결한다.

| 실패 | 원인 | v3 해결 |
|------|------|---------|
| 범용 태그 유사도 1.000 | "사회파 미스터리" 같은 태그 | 맥락 보존 reason 프롬프트 |
| 유저 피드백 변환 손실 | 피드백을 reason 형식으로 변환 | 피드백 원문 그대로 임베딩 |
| 벡터 합산으로 정보 손실 | reason+desc+genre 벡터를 가중평균 | Weighted MaxSim (각 점 독립 유지, 점 간 거리의 가중합으로 전환. 정보 손실 감소, 완전 제거는 아님) |
| Supabase IO 고갈 | DB에서 벡터 연산 | FastAPI + numpy 메모리 분리 |
| 장르 필터로 가둠 | genre include/exclude | 장르 벡터가 자연스럽게 거리 반영 |
| 스케일업 전 검증 안 함 | 3권으로만 테스트 | 212권 × 10 페르소나 검증 |
| LLM 생성 리뷰로 자기검증 | description 기반 가짜 리뷰 | 실제 독자 리뷰로 검증 |
