## 1. 핵심 원칙

**취향의 품질이 최우선이다.**

추천 엔진의 유일한 목적은 "유저가 다음에 읽을 책으로 이걸 추천받았을 때 '아 이거 내 취향이다'라고 느끼는 것"이다. 응답 시간, 확장성, 코드 구조는 모두 이 목적을 달성한 뒤에 의미가 있다.

### 1.1 취향이란 무엇인가

취향은 단일 속성이 아니라 **여러 벡터의 연결성으로 형성되는 공간**이다.

- 유저가 책에 남긴 **피드백**은 취향의 명시적 신호
- 유저가 좋아한 책의 **reason**(이 책을 좋아할 이유)은 취향의 축
- 책의 **description**은 전체 분위기
- 책의 **장르(L1/L2)**는 장르 공간에서의 위치 (중분류 + 소분류)

이 네 가지는 모두 같은 임베딩 공간(text-embedding-3-large, 2000차원)에 존재하는 **점(벡터)**이다. 각각이 독립적인 점으로 존재하고, 점들 사이의 거리(코사인 유사도)가 취향의 연결성을 나타낸다. 벡터 자체를 합치거나 평균내지 않는다 — 각 점은 공간에서 독립적으로 유지된다. 최종 추천 스코어를 만들 때 점 간 거리의 가중합이 사용되지만, 이는 벡터 공간에서의 정보를 보존한 상태에서의 수치화이다 (섹션 4.1 참조).

### 1.2 취향 품질의 스펙트럼

유저의 입력량과 구체성에 따라 추천 품질이 달라진다.

| 입력 상태 | 추천 품질 | 설명 |
|-----------|----------|------|
| 좋아요 1~2권 | 방향만 잡힘 | 비슷한 장르 위주, 변별력 약함 |
| **좋아요 3권** | 방향 확실 | 장르+분위기 매칭 시작 |
| **좋아요 5권** | 신뢰 가능 | 모든 취향 유형(소설/실용서/과학)에서 안정 |
| 좋아요 5권 + 싫어요 1권 이상 | 변별력 강화 | 싫어하는 영역이 확실히 밀림 |
| 좋아요 5권 + 구체적 피드백 2개 이상 | 정밀 추천 | 피드백이 취향 축을 날카롭게 잡아줌 |

**온보딩 최소 조건: 좋아하는 책 5권.** 이 조건부터 추천을 시작한다. 피드백과 싫어요는 자연스럽게 쌓이도록 유도한다.

### 1.3 피드백 깊이와 취향의 관계

같은 책에 대해서도 피드백 깊이에 따라 생성되는 취향 벡터가 다르다.

**구체적 피드백** ("사회의 구조적 폭력을 범죄 서사로 파고드는 시선이 날카로웠다")
→ 피드백 자체가 강한 취향 벡터. 유저가 이 책의 **어떤 축**을 좋아했는지 명확.
→ 해당 축과 가까운 다른 책이 정밀하게 매칭됨.

**피드백 없음 (호오만)**
→ 피드백 벡터 없음.
→ 책의 reason/desc/L1/L2 벡터만으로 취향 공간 형성.
→ 가장 넓은 추천. 피드백이 쌓일수록 좁혀짐.

참고: "좋았음", "별로" 같은 짧은 피드백도 임베딩되어 피드백 벡터로 사용된다. 정보량이 적어 넓은 매칭이 되지만, 피드백이 있는 것과 없는 것의 차이는 가중치 구조(섹션 3.2)에서 자동 반영된다. 구현에서는 **피드백 있음/없음** 2가지로만 분기한다.

**싫어요**
→ 역방향 벡터. 해당 영역의 책을 밀어냄.
→ 구체적 싫어요 피드백은 특정 축만 밀어냄, 호오만 싫어요는 책 전체 영역을 밀어냄.

### 1.4 책이 가진 벡터와 유저 피드백 벡터의 관계

책과 유저의 벡터는 **같은 공간에 공존**한다. 합치거나 변환하지 않는다.

```
같은 임베딩 공간 (2000차원):

  유저의 점들:
    ● 피드백 "사회문제를 파고드는 시선이 날카로웠다"
    ● 피드백 "캐릭터 관계가 리얼해서 몰입됐다"
    ○ 좋아한 책A의 reason "범죄 피해자 가족의 고통을 통해 사법 시스템의 모순을 드러냄"
    ○ 좋아한 책A의 desc 벡터
    ○ 좋아한 책A의 L1 벡터 (중분류)
    ○ 좋아한 책A의 L2 벡터 (소분류)
    ○ 좋아한 책B의 reason들...
    ✕ 싫어한 책C의 벡터들 (역방향)

  후보 책의 점들:
    ◆ reason "소외된 이웃의 삶을 담담하게 비추는 사회 고발적 시선"
    ◆ reason "다층 시점 전환으로 사건의 전모가 서서히 밝혀지는 구조"
    ◆ desc 벡터
    ◆ L1 벡터 (중분류)
    ◆ L2 벡터 (소분류)

  추천 = 유저의 점들과 후보 책 점들이 가까운 정도
```

유저의 피드백 벡터 ●가 후보 책의 reason 벡터 ◆과 가까우면 → 취향 매칭.
유저의 L1/L2 벡터 ○가 후보 책의 L1/L2 벡터 ◆과 가까우면 → 같은 장르 공간.
유저의 싫어요 벡터 ✕가 후보 책의 벡터와 가까우면 → 역방향으로 밀어냄.

**이 연결들이 동시에 가까운 책이 강추천, 하나만 가까운 책은 약추천.** 각 타입별로 독립적으로 계산된 뒤 가중합으로 스코어가 만들어진다 (섹션 4.1 참조).

## 2. 아키텍처

```
┌─────────────┐     ┌──────────────────────┐     ┌─────────────┐
│ Flutter 앱   │────▶│ FastAPI 추천 서버       │────▶│ Supabase    │
│             │◀────│                      │     │ (저장만)     │
└─────────────┘     │  ┌────────────────┐  │     └─────────────┘
                    │  │ numpy 벡터 매트릭스│  │
                    │  │ (메모리, BLAS)   │  │
                    │  └────────────────┘  │
                    └──────────────────────┘
```

- **Supabase**: Auth, 책 데이터, 유저 데이터, 피드백 저장. 추천 연산 하지 않음.
- **FastAPI 추천 서버**: 벡터 검색, 추천 계산. 서버 시작 시 전체 벡터를 메모리에 로드.
- **numpy 행렬곱**: 추천 스코어링은 reason avg_maxsim + desc/L1/L2 cosine + fb_desc cosine으로 구성되며, 각각 벡터 간 행렬곱이 필요하므로 단일 벡터 ANN(FAISS)으로 직접 할 수 없음. numpy BLAS 배치 행렬곱으로 처리. 2,000권 기준 ~20ms 추정. 규모 확장 시 MUVERA 스타일 압축 또는 pre-filtering 적용 가능.
- **Flutter 앱**: 온보딩, 피드백 입력, 추천 결과 표시.

### 2.1 왜 Supabase에서 추천하지 않는가

v1~v2에서 Supabase pgvector로 추천을 시도했으나:
- 2000차원 벡터 × 19,000건 CROSS JOIN → IO 고갈, 타임아웃
- Free tier Disk IO 한도에서 배치와 실시간이 공존 불가
- 유저 1,000명 동시 접속 시 DB가 버틸 수 없음

추천 연산을 별도 서버로 분리하면 Supabase는 저장만 담당하고, 벡터 검색은 numpy 행렬곱(메모리)으로 밀리초 단위 처리.

## 3. 데이터 모델

### 3.1 책 벡터 (사전 배치)

각 책은 다음 벡터를 가진다:

| 벡터 | 개수 | 생성 시점 | 설명 |
|------|------|----------|------|
| reason 벡터들 | 5~8개 | 배치 (LLM 추출 → 임베딩) | "이 책을 좋아할 이유" 각각의 벡터 |
| desc 벡터 | 1개 | 배치 (rich_description HTML 제거 후 최대 2000자 임베딩) | 책의 전체 분위기 벡터. 200자로는 마케팅 문구만 담기므로 2000자 필요 (테스트: 2000자가 정답1위 6/7, 200자는 5/7) |
| L1 벡터 (중분류) | 1개 | 배치 (중분류 텍스트 임베딩) | "소설/시/희곡", "경제경영" 등. 장르 대분류 구분 |
| L2 벡터 (소분류) | 1개 | 배치 (소분류 이하 텍스트 임베딩) | "한국소설", "재테크/투자 주식/펀드" 등. 세부 장르 구분 |

**장르 벡터를 L1/L2로 분리한 이유:**
기존 장르 문자열 전체(`국내도서>소설/시/희곡>한국소설>2000년대 이후`)를 통째로 임베딩하면 상위 분류가 같을 때 하위 분류 차이가 묻힌다. 예: 한국소설↔한국시 유사도 0.832 (구분 불가). L1/L2로 분리하면 L2에서 한국소설↔한국시 유사도가 0.450으로 벌어진다. 동시에 한국소설 내부(2000년대↔90년대)는 L2 유사도 0.970으로 여전히 가깝다.

**L1/L2 분리 규칙:**
```
원본: "국내도서>소설/시/희곡>한국소설>2000년대 이후 한국소설"
→ L1: "소설/시/희곡" (국내도서 제거 후 첫 번째 레벨)
→ L2: "한국소설 2000년대 이후 한국소설" (나머지 전부 이어붙임)
```
깊이가 3~5단계로 들쭉날쭉해도 일관되게 2개 레벨로 분리된다. L1은 중분류별로 고유 텍스트가 ~20개뿐이므로 동일 텍스트는 임베딩을 재사용한다.

**reason 추출 프롬프트 핵심 규칙:**
- 15~40자의 구체적 명사구/설명구
- 이 책만의 맥락이 담겨야 함 (다른 책에 붙여넣을 수 없어야 함)
- 고유명사, 평론 표현, 범용 태그 제외
- 예: "범죄 피해자 가족의 고통을 통해 사법 시스템의 모순을 드러냄" (O)
- 예: "사회파 미스터리" (X — 범용 태그)

### 3.2 유저 취향 벡터 (실시간 생성)

유저가 책에 대해 입력할 때마다 취향 벡터가 생성된다.

| 입력 유형 | reason 가중치 | 피드백 역할 |
|-----------|-------------|-----------|
| 호 + 피드백 있음 | reason: ±0.5 | 피드백 임베딩이 reason_score(×3.0)와 fb_desc_score(×2.0)에 참여 |
| 호 + 피드백 없음 | reason: ±1.0 | 피드백 없으므로 fb_desc_score = 0, reason만으로 판단 |
| 오 + 피드백 있음 | reason: -0.5 | 피드백 임베딩이 -3.0으로 역방향, fb_desc_score도 역방향 |
| 오 + 피드백 없음 | reason: -1.0 | reason만 역방향으로 밀어냄 |

**neutral(보통) 처리:** 추천 계산에서 제외. "보통"은 긍정도 부정도 아니므로 취향 벡터에 기여하지 않는다. 서재에는 기록되지만 추천 입력으로 사용하지 않음.

desc/L1/L2는 reason과 별도로 계산된다 (섹션 4.1 참조). 싫어요 책의 desc/L1/L2는 계산에서 제외된다.

**피드백 임베딩 생성:** 유저가 피드백 텍스트를 작성할 때 OpenAI embedding API 1회 호출 (~100ms) → DB에 저장. 추천 시에는 저장된 벡터만 사용하므로 추가 API 호출 없음.

**피드백의 영향:** 피드백이 있는 책은 reason_score에서 3.0 가중치 + fb_desc_score에서 후보 desc와 직접 비교로 작용한다. 이로 인해 피드백을 남긴 장르의 추천 비중이 권수와 무관하게 높아진다. 예: 경제 3권(호오만) + 소설 2권(구체 피드백) → 소설이 추천의 다수를 차지.

## 4. 추천 알고리즘

### 4.1 분리 계산 스코어링 (Separated Scoring with Feedback-Desc Bridge)

각 벡터 타입(reason, desc, L1, L2)을 독립적으로 계산한 뒤 가중합한다. 기존 ColBERT MaxSim의 포인트 클라우드 원칙(벡터를 합치지 않음)을 유지하면서, 타입별 분리 계산으로 reason 개수 편향을 제거하고 피드백이 장르 비중에 영향을 미치도록 한다.

```python
def recommend_score(liked_bids, fb_data, dislike_set, candidate_bid):
    """
    liked_bids: 유저가 좋아한/싫어한 책 ID 리스트
    fb_data: {book_id: {emb, is_dislike}} 피드백 임베딩
    dislike_set: 싫어요 book_id 집합
    candidate_bid: 후보 책 ID
    """

    # ── 1. reason_score: 가중 avg_maxsim ──
    # 피드백 있는 책: 피드백 emb(×3.0) + reason(×0.5)
    # 피드백 없는 책: reason(×1.0)
    # 싫어요: 부호 반전
    # → 전체 mean (reason 개수 편향 제거)

    # ── 2. desc_score: 좋아한 책 desc ↔ 후보 desc (max) ──
    # 싫어요 책은 제외

    # ── 3. L1_score: 좋아한 책 L1 ↔ 후보 L1 (max) ──
    # 싫어요 책은 제외

    # ── 4. L2_score: 좋아한 책 L2 ↔ 후보 L2 (max) ──
    # 싫어요 책은 제외

    # ── 5. fb_desc_score: 피드백 emb ↔ 후보 desc (mean) ──
    # 피드백이 있는 책에 대해서만
    # 좋아요 피드백: +1, 싫어요 피드백: -1

    return (1.0 * reason_score
          + 0.5 * desc_score
          + 3.0 * L1_score
          + 1.0 * L2_score
          + 2.0 * fb_desc_score)
```

**각 항목의 역할:**

| 항목 | 가중치 | 역할 | 계산 방식 |
|------|--------|------|----------|
| reason_score | ×1.0 | 취향 내용 매칭의 핵심 | 유저 reason/피드백 ↔ 후보 reason, avg_maxsim |
| desc_score | ×0.5 | 책 전체 분위기 매칭 보조 | 유저 desc ↔ 후보 desc, max cosine |
| L1_score | ×3.0 | 대분류 장르 구분 (어린이/성인, 소설/경제) | 유저 L1 ↔ 후보 L1, max cosine |
| L2_score | ×1.0 | 소분류 장르 구분 (한국소설/시, 뇌과학/물리) | 유저 L2 ↔ 후보 L2, max cosine |
| fb_desc_score | ×2.0 | **피드백↔후보 desc 직접 비교 (취향 전환의 핵심)** | 피드백 emb ↔ 후보 desc, mean cosine |

**기존 합산 MaxSim에서 변경된 이유:**

기존 방식(모든 벡터를 하나의 점 구름에 넣고 합산 MaxSim)의 두 가지 문제:

1. **reason 개수 편향**: 후보 책의 reason이 많을수록 MaxSim이 높아짐. 분리 계산 + avg로 제거.
2. **장르 잠금(lock-in)**: L1이 다수 장르를 독점하여 피드백으로 취향 전환이 불가. fb_desc_score가 피드백 내용으로 장르 비중을 자연스럽게 조절.

**fb_desc_score가 취향 전환을 해결하는 원리:**

피드백 텍스트("범죄의 이면에 숨겨진 인간 심리를 탐구하며...")는 소설 desc와 가깝고 경제 desc와는 멀다. 이 유사도 차이 × 2.0이 L1의 다수 장르 우위를 상쇄한다. 피드백이 없으면 fb_desc_score = 0이 되어 기존 공식과 동일하게 동작한다.

### 4.2 왜 이 방식이 작동하는가

**어린이 과학책이 성인 과학 독자에게 추천되지 않는 이유:**
- L1에서 "어린이" vs "과학"은 유사도 0.290으로 매우 멀다 (×3.0 가중치)
- reason에서 "진화론"이 겹쳐도 L1 갭 × 3.0을 넘지 못한다

**한국소설 독자에게 시가 추천되지 않는 이유:**
- L1은 둘 다 "소설/시/희곡"으로 동일 (유사도 1.0)
- 하지만 L2에서 "한국소설" vs "시 한국시"는 유사도 0.450으로 벌어진다
- reason 내용도 서사 vs 서정으로 다르므로 reason_score에서도 차이

**취향 전환(경제 3권 + 소설 2권, 소설에 피드백)이 반영되는 이유:**
- L1은 경제(3권) max = 1.0 > 소설 max = ~0.3 → 경제 유리
- 하지만 소설 피드백의 fb_desc_score가 소설 후보의 desc와 높은 유사도 → 소설 후보의 총점 상승
- 피드백의 깊이가 장르 비율을 결정: 구체적 피드백일수록 해당 장르 비중이 높아짐
- 피드백 없이 호오만이면 읽은 권수 비율이 자연스럽게 반영됨

## 5. reason 추출 프롬프트

```
이 책을 좋아할 독자 관점에서 "좋아할 이유" 5~8개. 15~40자 명사구.
이 책만의 구체적 맥락. 고유명사/평론/범용태그 제외.

도서: {title} ({genre})
설명: {desc}  ← rich_description HTML 제거 후 최대 1500자

JSON: {"reasons": [...]}
```

**프롬프트 검증 기준:**
- 같은 reason이 다른 책에 그대로 붙여넣을 수 있으면 실패
- "성장", "감동", "몰입감" 같은 범용 태그 → 실패
- "범죄 피해자 가족의 고통을 통해 사법 시스템의 모순을 드러냄" → 통과

**검증 결과:**
- 6권 → 14권 → 85권 → 212권 단계적 테스트
- 유사도 1.000 문제(범용 태그) 완전 해결
- 유사도 그라데이션 0.3~0.6 범위에서 자연스러운 분포

### 6.1 FastAPI 추천 서버

```
recommendation-server/
```

**엔드포인트:**

| 엔드포인트 | 메서드 | 인증 | 설명 |
|-----------|--------|------|------|
| `GET /recommend/{user_id}` | GET | JWT (본인만) | 취향 기반 추천 목록 반환 (top N, 기본 10) |
| `GET /similar/{book_id}` | GET | JWT | book-to-book 유사도 (top N, 기본 10). desc 코사인 유사도 단일. 책 상세 "비슷한 책" 섹션용. |
| `POST /feedback` | POST | JWT | 피드백 수신 → 임베딩 → Supabase 저장 → 메모리 업데이트 |
| `GET /health` | GET | 없음 | 서버 상태 + 로드된 벡터 수 |

**요청/응답 스키마:**

```python
# GET /recommend/{user_id}?limit=10
# Response:
{
  "user_id": "uuid",
  "recommendations": [
    {"book_id": "uuid-string", "score": 7.42, "title": "...", "author": "...", "cover_url": "..."},
    ...
  ],
  "meta": {"total_liked": 5, "total_disliked": 1, "has_feedback": true}
}

# GET /similar/{book_id}?limit=10
# Response:
{
  "book_id": "uuid-string",
  "similar": [
    {"book_id": "uuid-string", "score": 0.87, "title": "...", "author": "...", "cover_url": "..."},
    ...
  ]
}

# POST /feedback
# Request (user_id는 JWT에서 추출, body에 넣지 않음):
{
  "book_id": "uuid-string",
  "rating": "good",         # "good" | "neutral" | "bad"
  "review_text": "...",      # optional
  "emotion_tags": ["여운", "따뜻한"]  # optional
}
# Response:
{"status": "ok", "feedback_id": "uuid"}
```

**인증:**

Flutter 앱이 Supabase에서 받은 JWT 토큰을 `Authorization: Bearer <token>` 헤더로 전달. FastAPI에서 Supabase의 JWT secret으로 검증. `/recommend/{user_id}`는 토큰의 user_id와 경로 user_id 일치 확인.

**서버 시작 시:**
1. Supabase에서 전체 book_love_reasons + book_v3_vectors + books 테이블 로드
2. reason/desc/L1/L2 벡터를 numpy 행렬로 메모리에 적재
3. book-to-book 유사도용 desc 행렬 별도 구축

**피드백 수신 시:**
1. 피드백 텍스트 → OpenAI embedding API (1회, ~100ms)
2. Supabase에 피드백 저장 (user_books 테이블)
3. 메모리 내 유저 취향 벡터 업데이트 (다음 추천에 즉시 반영)

**에러 처리:**

| 상황 | 응답 |
|------|------|
| 유저의 좋아요 책이 0권 | 200 + 빈 배열 + meta.total_liked=0 |
| 존재하지 않는 book_id | 404 |
| JWT 검증 실패 | 401 |
| user_id 불일치 | 403 |
| OpenAI API 실패 (피드백) | 200 + 피드백 텍스트는 저장, 임베딩만 누락 (배치에서 재시도) |

**벡터 갱신 전략:**

| 트리거 | 방식 |
|--------|------|
| 피드백 수신 | 즉시 메모리 업데이트 (유저 취향 벡터만) |
| 배치에서 새 책 추가 | 서버 재시작 또는 `POST /admin/reload` (API key 인증) |
| MVP 규모 (2,500권) | 서버 재시작으로 충분 (~5초). 핫 리로드는 나중에. |
