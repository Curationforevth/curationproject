## 핵심 원칙 (Eden이 반복 강조)

### 1. "왜 좋아했는가"가 취향을 결정한다
- 책을 좋아한다는 사실보다 **왜 좋아했는지**가 추천을 좌우
- "심리묘사가 좋았다"가 추리소설에 대한 건지 로맨스에 대한 건지에 따라 의미가 완전히 다름
- 피드백은 반드시 **그 책의 맥락 안에서** 해석돼야 함

### 2. 유저의 피드백이 유의미하게 활용돼야 한다
- 피드백을 단순 가중치(weight)로만 쓰면 안 됨 — 내용이 반영돼야 함
- 감성태그 10개에 한정하면 확장성 없음 — 자유 텍스트 리뷰도 활용해야
- 입력할수록 추천이 **눈에 보이게** 변해야 (카피가 아닌 실제 UX에서 체감)

### 3. 확장성
- 유저가 늘어나도 비용이 선형 증가하면 안 됨
- 유저당 매번 LLM 호출하는 구조는 비확장적
- 책당 한 번(배치) 처리하는 것은 확장적

## MVP 추천 2가지 (확정)

### 1. 책 상세 → "비슷한 책" (book-to-book)
- 작동 조건: 없음. 책 1권만 있어도 작동
- 로직: 해당 책의 임베딩 벡터로 pgvector 코사인 유사도 → Top-N
- RPC: `match_books_by_similarity` (구현 완료)

### 2. 서재 메인 → "추천 섹션" (taste-to-book)
- 작동 조건: 추천 신뢰도 스코어 충족
- 로직: 취향 벡터 기반 추천 + 속성 매칭 재정렬
- 다중 취향 벡터 지원 (K-means 클러스터링 시)

## 아키텍처 (확정)

### 두 경로 구조

| 경로 | 방식 | 용도 |
|------|------|------|
| **즉시** | Supabase RPC | book-to-book 유사도, 취향 벡터 재계산, 신뢰도 스코어 |
| **배치** | GitHub Actions Python | 전체 유저 취향 벡터 갱신, K-means 클러스터링, 책 속성 추출 |

### 즉시 경로 — 유저 피드백 시
```
피드백 제출 → RPC: recompute_taste_vector_immediate
  → method 확인 (weighted_avg / kmeans)
  → 취향 벡터 즉시 재계산
  → 신뢰도 스코어 갱신
```

### 배치 경로 — 정교한 분석
```
daily-taste-recompute (KST 07:00):
  → 전체 유저 취향 벡터 재계산
  → 충분한 데이터: K-means 클러스터링
  → 신규 책: 속성 추출
```

## 피드백 반영 — 실험 결과 및 현재 방향

### 시도한 접근과 결과

| # | 접근 | 결과 | 문제 |
|---|------|------|------|
| 1 | 책 임베딩 가중 평균 (피드백=weight만) | 유사도 0.63+ | **피드백 내용이 반영 안 됨** — 같은 책에 다른 피드백을 줘도 추천이 동일 |
| 2 | 피드백 텍스트 직접 임베딩 | 유사도 0.49 | **벡터 공간 불일치** — 경험적 텍스트와 설명적 텍스트가 다른 공간 |
| 3 | β 혼합 (책 임베딩 × β + 피드백 임베딩 × (1-β)) | 효과 미약 | 피드백 임베딩의 영향력이 너무 작음 |
| 4 | 감성태그 방향 벡터 (α=0.5) | **작동** — Top-10에서 4개 차이 | 태그 10개에 한정, 자유 텍스트 못 씀 |
| 5 | LLM 피드백 번역 | 미실험 | 할루시네이션 위험 (유저가 안 말한 걸 추가) |
| 6 | 피드백 임베딩→가까운 책→방향 벡터 | 미실험 | "심리묘사" 방향이 맥락 무관하게 동일 |
| 7 | **구조화된 속성 추출 + 속성 매칭** | **유력** — 초기 실험 긍정적 | 추가 실험 필요 |

### 현재 유력한 접근: 구조화된 책 속성

**핵심 아이디어:** 각 책의 특성(심리묘사, 서사구조, 긴장감 등)을 구조화된 점수로 미리 추출해두고, 유저 피드백에서 어떤 속성을 좋아하는지 파악하여 매칭.

```
[책 속성 추출 — 배치, 책당 1회]
책 설명/소개 → LLM → {서사구조: 0.9, 심리묘사: 0.8, 세계관: 0.9, ...}

[유저 피드백 → 속성 매핑]
"스토리가 탄탄하고 심리묘사가 좋았다" → {서사구조: 0.8, 심리묘사: 0.9}

[추천]
1단계: 임베딩 유사도로 후보 추출 (장르/주제 유사)
2단계: 유저 속성 프로필 vs 후보 책 속성 매칭으로 재정렬
최종 점수 = 임베딩 유사도 × (1 + α × 속성 매칭 점수)
```

**왜 이게 나은가:**
- 책당 1회 추출 (유저당 X) → 확장적
- 자유 텍스트도 속성으로 매핑 가능
- 임베딩은 "어떤 종류의 책인지", 속성은 "이 책의 어떤 점이 좋은지" → 역할 분리
- 기존 DB 데이터(설명, 소개, 키워드)를 더 잘 활용

**초기 실험 결과 (2026-03-26):**
- 속성 추출 품질: 좋음 (드래곤라자 → 세계관=1.0, 철학적깊이=0.9, 서사구조=0.8)
- 피드백→속성 매핑: 좋음 ("스토리 탄탄+심리묘사" → 서사구조=0.8, 심리묘사=0.9)
- 추천 결과: 속성 점수 높은 책이 위로 올라옴 (심리묘사=0.9인 책이 상위)
- **문제:** 테스트가 카카오 즉석 임베딩이라 후보 품질이 낮았음 → DB 정식 책으로 재실험 필요

### 속성 목록 (초안, 15개)

서사구조, 심리묘사, 세계관, 문체미, 유머, 속도감, 긴장감, 감정적깊이, 사회비평, 철학적깊이, 분위기, 독창성, 몰입감, 캐릭터매력, 여운

### 다음 세션에서 해야 할 것

1. **DB에 있는 유명한 책으로 재실험** — Eden이 아는 책 3~5권 골라서 피드백 → 추천 결과를 Eden이 직접 판단
2. **속성 추출 배치 파이프라인 설계** — 기존 enrichment 파이프라인에 속성 추출 스텝 추가
3. **속성 매칭 가중치(α) 튜닝**
4. **추천 신뢰도 임계값 재실험** — 속성 매칭이 추가되면 기존 실험 결과가 달라질 수 있음

## 추천 신뢰도 스코어 (구현 완료, 임계값 미정)

### 입력 변수
```
confidence_score = f(
  feedback_depth_score,    # 피드백 깊이 합산
  genre_diversity,         # 장르 다양성
  rating_variance,         # 호오 분산
  book_count,              # 절대 권수
)
```

### 실험 결과 (390회, 2026-03-26)

| 유사도 구간 | 장르 일치율 | 판단 |
|---|---|---|
| 0.70+ | 96.7% | 매우 정확 |
| 0.65~0.70 | 78.2% | 좋음 |
| 0.60~0.65 | 65.6% | 쓸만함 |
| 0.55~0.60 | 60.4% | 애매 |
| 0.50 이하 | 48.1% | 의미 없음 |

**발견:**
- confidence 스코어 단독으로는 추천 품질 판단 불충분
- **유사도 점수 자체가 품질 필터링 기준이 돼야 함** (min_similarity ≥ 0.6 권장)
- 피드백 깊이보다 **책 권수**가 유사도에 더 큰 영향 (단, 피드백 내용 반영이 안 되는 현재 구조에서의 결과)
- 속성 매칭 도입 후 재실험 필요

## 취향 벡터 단계적 진화 (구현 완료)

| 단계 | 조건 | 방식 |
|---|---|---|
| 초기 | 1~10권 | 가중 평균 (RPC: 즉시, Python: 배치) |
| 중기 | 10~20권 | K-means (k=2~3) 시도, 실루엣 스코어 검증 |
| 후기 | 20권+ | K-means (k=2~5) + LLM 라벨 생성 |

### Bad 책 처리
- 취향 벡터 계산에서 제외 (weight=0)
- 추천 결과에서 bad 책과 유사도 높은 책(0.85+) 순위 하향

### 시간 가중치 (미구현)
```
final_weight = base_weight × recency_factor
recency_factor = max(0.5, 1.0 - (days_since_feedback / decay_period))
```
