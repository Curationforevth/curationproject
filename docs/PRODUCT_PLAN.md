# Curation Project — Product Plan

> 컨텐츠 소비 피드백 기반 취향 분석 & 추천 앱
> MVP: 도서 카테고리로 시작

---

## 1. 프로덕트 비전

**"베스트셀러만 읽는 시대에, 나만의 취향을 찾아가는 즐거움"**

사용자가 읽은 책에 대한 피드백을 남기면, 취향을 분석하고 취향에 맞는 새로운 책을 추천해주는 앱.

- 장기 목표: 도서 → 영화, 뮤지컬, 전시 등 전 카테고리 확장
- 타겟: 자기 취향을 아직 잘 모르는 일반 유저
- 플랫폼: iOS + Android 동시 출시

---

## 2. 핵심 가치

| # | 가치 | 설명 |
|---|------|------|
| 1 | **서재 경험** | 책을 하나씩 꽂아가는 뿌듯함. "한 권 더 꽂고 싶다"는 리텐션 루프 |
| 2 | **취향 발견** | 쌓인 피드백으로 "나는 이런 독자구나"를 알게 되는 경험 |
| 3 | **맞춤 추천** | 베스트셀러가 아닌, 나의 취향에 맞는 책 발견 |

> 서재는 추천을 위한 데이터 수집 수단이지만, 그 자체로 다른 서재 앱과 차별화된 감성적 경험을 제공해야 한다.

## 3. 로드맵

### Phase 1 — 서재 + 피드백 + 추천 (MVP)

- 책 검색 & 등록 (카카오 API 메인 + 알라딘 보완)
- 서재 UI — 커버 피드(기본) + 서가 뷰(전환) 하이브리드
- 책등 자동 생성 (표지 dominant color 추출 + 무드 기반 폰트 배정)
- 피드백 수집 (가이드 질문 + 자유 텍스트)
- 온보딩 플로우
- 드래그 정렬 (서가 뷰)
- 마일스톤 배경 변화
- **추천 엔진** — 데이터 양에 따라 자동 진화하는 구조를 MVP부터 셋업
  - 책 상세 → "비슷한 책" (book-to-book pgvector 유사도, 항상 작동)
  - 서재 메인 → "추천 섹션" (취향 벡터 기반, 신뢰도 스코어 충족 시 활성화)
  - 즉시 경로: 피드백 제출 → RPC로 취향 벡터 즉시 재계산 → 추천 즉시 반영
  - 배치 경로: 전체 유저 취향 벡터 재계산 + 클러스터링 (GitHub Actions)
  - 추천 신뢰도 스코어: 피드백 깊이/장르 다양성/호오 분산 종합 평가
  - 취향 벡터 단계적 진화: 가중 평균 → K-means 클러스터링 (데이터 충분 시 자동 전환)
- **스펙:** `docs/superpowers/specs/2026-04-01-recommendation-engine-v3-design.md`

### Phase 2 — 취향 프로필 + 추천 고도화

- 쌓인 피드백을 LLM으로 분석하여 "취향 요약" 제공
- 예: "당신은 캐릭터 성장 서사를 좋아하는 독자예요"
- 클러스터별 LLM 라벨 자동 생성 ("SF 세계관", "잔잔한 에세이")
- 추천 이유 표시 ("이 책의 서정적인 문체가 마음에 드실 거예요")

### Phase 3 — 카테고리 확장

- 도서 → 영화, 뮤지컬, 전시 등 전 카테고리 확장
- 카테고리 간 취향 연결 (책 취향 → 영화 추천 등)

## 4. MVP 핵심 유저 플로우

### 4-1. 온보딩 (목표: 3분 이내)

```
유명 책 표지 그리드 노출
   → "이 중에 읽은 책 있어?" 탭탭탭 선택
   → 선택할 때마다 서재에 꽂히는 애니메이션
   → 5권 이상 선택 시
   → "제일 좋았던 책 하나만 골라볼까?"
   → 가이드 질문 1~2개로 간단한 피드백
   → "당신의 서재가 시작됐어요" + 서재 화면
```

레퍼런스:
- **Spotify** — 아티스트 선택 시 버블 커지는 애니메이션, 선택의 쾌감
- **Pinterest** — 관심사 탭탭탭, 실시간 피드 변화
- **Letterboxd** — 영화 포스터 기반 별점 온보딩, 시각적 풍성함
- **Duolingo** — 온보딩 직후 "첫 수업 완료!" 즉시 성취감

### 4-2. 책 등록

```
책 검색 (제목/저자)
   → DB 매칭 (카카오/알라딘)
   → 표지에서 dominant color 2~3개 추출
   → 무드 기반 폰트 자동 배정
   → 서재에 추가 (책등 애니메이션)
   → 피드백 입력 유도
```

### 4-3. 피드백 입력

> **간소화 확정 (IA 스펙 정본):** 아래 플로우로 대체.

```
평가 (👍 좋았어요 / 🤔 보통이에요 / 👎 별로였어요)
   → 감정 태그 선택 (캐릭터 / 문체 / 세계관 / 스토리 / 메시지 / 분위기)
   → 한 줄 감상 (선택, 자유 텍스트)
   → [완료] 또는 [나중에 할게요] 스킵 가능
```

> "읽었어요" 직후 바로 피드백 화면 진입. 스킵하면 서재에 등록만 되고 피드백은 홈 "내 책" 섹션에서 나중에 유도.
> 별로였던 점은 별도 수집하지 않음 (평가에서 👎 선택 시 태그+감상으로 충분).

## 5. 서재 UI 설계

### 5-1. 앱 구조 (IA 스펙으로 대체)

> **주의:** 탭 구조, 피드 설계, 등록/피드백 플로우는 `docs/superpowers/specs/2026-04-03-app-ia-design.md`가 정본.
> 아래 5-2(커버 피드 카테고리)의 피드 행 구성 로직도 IA 스펙의 홈 섹션 설계로 대체됨.

**변경 요약:**
- 4탭(서재/탐색/추천/프로필) → **3탭(홈/+/서재)** + 상단 아바타(취향)
- 서재: 커버 피드 + 서가 뷰 토글 → **서가 뷰 단독** (커버 피드 제거)
- 큐레이션 피드: 서재 내 → **홈 탭으로 분리**
- 피드백: 등록과 동시 수집 → **등록 후 별도 유도 (스킵 가능)**

### 5-2. 커버 피드 — 카테고리 설계

서재 메인 화면은 **단순 목록이 아니라, 카테고리별 가로 스크롤 행으로 구성된 피드**다.
넷플릭스처럼 행이 곧 카테고리이며, 데이터가 쌓일수록 행이 풍성해진다.

#### 책의 상태

| 상태 | 의미 | 피드백 |
|------|------|--------|
| **읽은 책** | 핵심 상태. 피드백을 남길 수 있는 책 | 있음 / 유도 |
| **읽는 중** | 아직 다 안 읽은 책. 보조 상태 | 없음 |

> "읽고 싶은 책"은 MVP에 포함. 홈 큐레이션 북마크 + 책 상세에서 저장. 상세: `docs/superpowers/specs/2026-04-03-app-ia-design.md`

#### 피드 행 구성 로직

피드의 행은 **Phase에 따라 점진적으로 풍성해진다.**

**Phase 1 (MVP) — 메타데이터 + 추천**

| 행 | 조건 | 로직 |
|---|---|---|
| 읽는 중 | 읽는 중인 책이 1권 이상 | 고정 행 |
| 피드백을 남겨보세요 | 피드백 없는 읽은 책이 있을 때 | 피드백 유도 CTA (읽는 중 바로 아래) |
| **이 책은 어때요?** | 추천 신뢰도 스코어 충족 | 취향 벡터 기반 추천 (데이터에 따라 가중 평균 → 클러스터 자동 전환) |
| [작가명]의 책들 | 같은 작가 책이 2권 이상 | 자동 그룹핑 |
| [무드 라벨] | 같은 무드 태그 책이 5권 이상 | 책 등록 시 카카오 API `contents`(줄거리)를 LLM으로 분석 → 무드 태그 2~3개 자동 부여 (예: "잔잔한", "몰입감 있는", "따뜻한", "묵직한", "유쾌한") |

> 무드 태그는 Phase 1에서 메타데이터 기반으로 자동 부여하고, Phase 2에서 피드백 데이터가 쌓이면 정확도를 높인다.
> 추천 엔진 상세: `docs/superpowers/specs/2026-04-01-recommendation-engine-v3-design.md`

**Phase 2 (취향 프로필 + 추천 고도화) — LLM 분석 기반**

| 행 | 조건 | 로직 |
|---|---|---|
| [클러스터 라벨] 좋아하시니까 | 클러스터링 활성화 | 취향 클러스터별 추천 + LLM 라벨 |
| 나의 취향 요약 | 클러스터링 활성화 | LLM이 취향 요약 생성, 카드로 표시 |
| "몰입감 있는 이야기들" | 클러스터링 활성화 | 클러스터별 라벨 자동 생성 |

> 클러스터 라벨은 LLM이 피드백 내용을 분석해 자동 생성한다.
> 예: 유저가 "몰입감", "빠른 전개"를 자주 언급 → "페이지 터너" 카테고리 생성

#### 피드 개인화 원칙

1. **모든 유저에게 같은 구조, 다른 내용** — 행의 종류는 동일하나 내용은 각자의 데이터 기반
2. **데이터가 쌓이면 행이 늘어남** — 처음엔 2~3행, Phase 2에서 5~7행, Phase 3에서 추천 행 추가
3. **의미 없는 카테고리는 만들지 않음** — 등록 시기별, 출판사별 같은 유저에게 무의미한 분류 배제
4. **카테고리 라벨은 자동 생성** — 피드백 임베딩 클러스터의 대표 키워드를 LLM이 자연어 라벨로 변환

### 5-3. 서가 뷰 — 내가 꾸미는 서재

다른 도서 앱에 없는 **유일한 차별점**. 커버 피드가 자동 큐레이션이라면, 서가 뷰는 **직접 정리하는 감성 경험**.

#### 역할 분리

| 커버 피드 | 서가 뷰 |
|---|---|
| 자동 큐레이션 (무드/작가별) | 내가 직접 정리 |
| 탐색·관리 | 감상·커스터마이징 |
| 섹션별 가로 스크롤 | 전체 컬렉션을 선반에 나열 |

#### 서가 뷰 구성

- **전체 책을 선반에 플랫 나열** — 큐레이션 섹션 없음 (그건 커버 피드의 역할)
- 한 선반에 5~7권, 책이 늘면 선반이 아래로 추가
- **롱프레스 → 드래그로 위치 이동** (같은 선반 내 순서 변경, 선반 간 이동)
- 마일스톤 배경 변화가 이 뷰에서 체감됨

#### 책등 디자인 — 컬러 블록 방식

책등은 스큐어모피즘(나무 질감, 포일 등)을 쓰지 않는다. **표지의 대표색을 컬러 블록으로 분할 배치**하는 모던한 접근.

| 요소 | 설명 |
|------|------|
| **색상 추출** | `palette_generator`로 표지 이미지에서 dominant color 2~3개 추출 → DB 저장 |
| **컬러 블록 배치** | 추출된 색상을 그라데이션이 아닌 **영역 분할**로 배치 (60% 주색 / 30% 보조색 / 10% 액센트 띠) |
| **폰트 자동 배정** | 5~6개 한국어 폰트 풀 준비 (명조, 고딕, 라운드 등). 책 메타데이터(장르/무드)에 따라 LLM이 적합한 폰트 선택 |
| **세로 텍스트** | 제목은 주색 블록에, 저자는 보조색 블록에 배치. 가독성 최우선 (텍스트 잘림 없음) |
| **꽂히는 애니메이션** | 책 추가 시 "톡" 하고 꽂히는 모션 |
| **드래그 정렬** | 롱프레스 → 드래그로 책 위치 이동 (MVP 필수) |
| **선반** | 나무 질감 없이 심플한 라인 (3px, warm gray) |

##### 폰트 풀 (예시)

| 폰트 | 느낌 | 배정 기준 |
|------|------|-----------|
| Nanum Myeongjo | 문학적, 클래식 | 순수문학, 고전 |
| Noto Serif KR | 무겁고 진지한 | 역사소설, 논픽션 |
| Black Han Sans | 강렬, 임팩트 | 스릴러, 사회고발 |
| Gowun Batang | 우아, 섬세 | 시집, 에세이 |

| Gaegu | 캐주얼, 손글씨 | 판타지, 동화적 |

> 폰트 배정은 책 등록 시 LLM이 메타데이터(제목, 장르, 줄거리)를 보고 자동 선택. 유저가 직접 변경할 수 있는 기능은 추후 검토.

### 5-4. 마일스톤 — 서재 성장

서재 크기가 아니라 **배경/분위기가 변화**한다. 서가 뷰와 커버 피드 모두에 적용.

| 권수 | 테마 | 분위기 |
|------|------|--------|
| 0~9권 | 순백 #FFFFFF (기본 배경) | 깨끗한 시작 |
| 10~29권 | 우드톤 + 은은한 조명 | 나만의 공간 |
| 30~49권 | 짙은 라이브러리 톤 + 벽면 텍스처 | 본격 독서가 |
| 50~99권 | 다크 우드 + 골드 액센트 | 자부심 |
| 100권+ | 풀 다크 라이브러리 + 특별 이펙트 | 궁극의 서재 |

> Flutter `ThemeData`를 마일스톤별로 전환하여 구현.

## 6. 취향 모델링 — 벡터 유사도 방식

### 왜 이 방식인가

기존 방식의 한계:
- 책의 특징(태그)을 사람이 미리 정의해야 함
- 새로운 피드백 유형이 생기면 분류 체계 수정 필요
- 취향 간 correlation 공식을 직접 설계해야 함

벡터 유사도가 해결하는 것:
- 분류 체계 없이 자연어를 그대로 벡터(숫자)로 변환
- "캐릭터 성장이 좋았어" ≈ "주인공의 변화가 인상적" → 자동으로 유사한 벡터
- 취향 간 관계도 벡터 공간에서 자연스럽게 형성

### 구조

```
[유저 피드백]              [책 메타데이터]
자유 텍스트 + 가이드 응답    줄거리 + 리뷰 + 장르 + 키워드
         ↓                          ↓
     LLM embedding              LLM embedding
         ↓                          ↓
     유저 취향 벡터              책 특징 벡터
              ↓        ↓
           코사인 유사도 계산
                ↓
          "이 책 어때요?"
```

### 설계 포인트

- **취향 벡터는 여러 개**: 한 사람이 SF도 에세이도 좋아할 수 있음. 클러스터링으로 취향 축 분리
- **피드백 품질 확보**: 가이드 질문으로 구조화된 데이터 + 자유 텍스트 조합
- **Negative signal 활용**: "별로였던 점"으로 해당 방향의 책 필터링

### 콜드 스타트 대응

- 책 1권만 있어도 "비슷한 책" 추천 가능 (book-to-book pgvector 유사도)
- 추천 신뢰도 스코어가 임계값 넘으면 서재 메인에 취향 기반 추천 섹션 활성화
- 신뢰도 미달 시 프로그레스 안내 ("감성태그를 남기면 추천이 더 정확해져요")
- 상세: `docs/superpowers/specs/2026-04-01-recommendation-engine-v3-design.md`

## 7. 기술 스택 (제안)

| 영역 | 스택 | 이유 |
|------|------|------|
| **앱 (프론트)** | Flutter | iOS+Android 크로스플랫폼, 서재 애니메이션 구현에 강함, 코드베이스 하나로 효율적 |
| **백엔드 인프라** | Supabase | 인증/DB/스토리지 올인원, 초기 서버 관리 부담 없음 |
| **책 데이터** | 카카오 책 검색 API (메인) + 알라딘 API (보완) | 카카오: 실시간 검색, 알라딘: 베스트셀러 배치 로드 |
| **임베딩** | OpenAI text-embedding-3-small | 비용 저렴, 품질 충분 |
| **벡터 저장** | Supabase pgvector | 별도 벡터 DB 없이 Supabase 내에서 해결 |
| **추천 (실시간)** | Supabase RPC (pgvector) | 유사도 검색 + 취향 벡터 재계산, DB 내부 처리 |
| **추천 (배치)** | Python + GitHub Actions | K-means 클러스터링 + LLM 취향 요약 (기존 파이프라인 패턴) |

> 이 스택은 제안이며, 개발자 리뷰 후 확정 필요.
