# ADR-001: 추천 엔진 — 벡터 유사도 방식 채택

- **날짜**: 2026-03-18
- **상태**: 제안 (개발자 리뷰 필요)
- **참여자**: Eden (PM)

---

## 배경

사용자가 책에 대해 남긴 피드백에서 취향을 파악하고, 취향 간 관계를 계산해서 맞춤 추천을 제공해야 한다.

## 검토한 대안

### A. Content-based filtering (태그 기반)

책마다 특징 태그를 미리 정의하고, 유저 피드백으로 태그별 가중치를 쌓는 방식.

- 장점: 유저 한 명 데이터만으로 동작
- 단점: 태그 체계를 사람이 설계해야 함. "왕좌의 게임 캐릭터 특성"을 어떤 축으로 분류할지 정의가 어려움. 새 피드백 유형 생기면 체계 수정 필요.

### B. Collaborative filtering (넷플릭스 방식)

비슷한 취향 유저들이 좋아한 책을 추천하는 방식.

- 장점: 태그 정의 불필요, 스케일 나면 강력
- 단점: 유저가 적을 때 작동 안 함 (cold start). MVP 단계에서는 사실상 불가능.

### C. LLM 임베딩 + 벡터 유사도 (채택)

유저 피드백과 책 메타데이터를 LLM으로 임베딩하고, 벡터 공간에서 코사인 유사도로 추천.

- 장점: 태그 체계 불필요, 자연어 피드백이 곧 취향 벡터, 취향 간 관계가 자동으로 형성
- 단점: 임베딩 API 비용 (단 text-embedding-3-small 기준 매우 저렴), LLM 의존성

## 결정

**C안 (벡터 유사도) 채택.**

핵심 이유:
1. 태그 체계를 미리 설계할 필요 없음 → PM/개발 모두 초기 부담 감소
2. 자연어 피드백을 그대로 활용 → 유저 입력의 자유도 높음
3. 2인 팀에서 관리 가능한 복잡도

## 리스크 & 미결사항

- 임베딩 품질이 추천 품질을 좌우 → 프롬프트/모델 선택 실험 필요
- 취향 벡터 클러스터링 방식 미정 (한 유저의 다중 취향 축 관리)
- Phase 3에서 실제 구현 시 개발자 주도로 설계 확정 필요
